# CMDB Lite PRD v3 建议清单

> **文档版本**：v3 建议稿（待评审）
> **生成日期**：2026-05-20
> **视角**：网络安全产品经理
> **基准**：`docs/CMDB-Lite-PRD-v2.md` + `docs/CMDB-Lite-差异分析-v1.md`
> **目标读者**：产品决策人 / Kiro 后续开发执行者
> **使用方式**：本文是基于 v0.1 现状盘查后的 v3 增强建议清单。每条建议包含「动机」「改造点」「优先级」三段式描述。待产品决策人 review 后决定是否纳入正式 PRD v3。

## 0. 文档说明

### 0.1 现状回顾

截至 2026-05-20，v0.1 已完成差异分析中的 P0/P1/P2 主要项（详见 `CMDB-Lite-差异分析-v1.md`）：

- ✅ 资产 CRUD、nmap 导入、扫描差异确认（ScanConfirm 三栏页）
- ✅ 三权分立 + 审计日志 + 不可变性触发器
- ✅ drawio iframe 真实集成 + LLM 路由
- ✅ 安全报表（端口暴露、危险端口、影子资产）
- ✅ 系统配置真正生效 + LLM API Key 加密

### 0.2 v3 建议的总体定位

v0.1 让产品**功能闭环**，v3 的目标是让产品**有差异化卖点**。三条主线：

1. **合规力**：从"日志能查"升级到"等保审计员看得懂、签得了字"
2. **安全力**：从"通用 CMDB + 报表"升级到"轻量级风险面板"
3. **工程力**：补齐速率限制、备份、SBOM、可观测性等"不性感但不能少"的基线

### 0.3 优先级符号

- **P0**：差异化核心，强烈建议纳入 v3 正式范围
- **P1**：合规/安全闭环必需
- **P2**：体验加分项，可放 v3.1
- **P3**：工程基线，与功能并行推进

## 1. 合规与等保（用户最在意的核心场景）

### 1.1【P0】等保 2.0 控制项映射表

**动机**
审计员现场会按控制项条目核查（8.1.4.1 身份鉴别、8.1.4.2 访问控制、8.1.4.3 安全审计……），目前审计页只能给一份"流水日志"，缺少"按控制项归类"的视图。这是国央企客户最敏感的痛点。

**改造点**
- 在 audit 模块新增 *Compliance Mapping* 视图
- 把 `audit_logs.action_type × target_type` 映射到等保 2.0 三级关键控制项
- 提供"控制项符合性报告 PDF"导出（按控制项分组聚合日志）
- 新建 `compliance_mappings` 表，存放映射规则（可由 super_admin 配置）

**比纯流水日志更有说服力，是销售场景下的强差异化点**

**预估工作量**：3 人天（后端 1.5 + 前端 1 + 文档 0.5）

---

### 1.2【P1】审计日志完整性签名（防抵赖）

**动机**
当前虽然加了 SQLite 触发器禁止 UPDATE/DELETE（migration `15ea56d0ba98`），但绕过应用层（直接 sqlite3 命令行）仍可改。等保高级别要求"防止抵赖"，纯触发器不够。

**改造点**
- 新建 `audit_logs.chain_hash` 字段
- 入库前用 HMAC-SHA256（密钥放 `secrets/audit_chain_key.txt`）对 `[prev_hash + timestamp + user_id + action_type + target_type + details]` 算 chain hash
- 后台提供 `POST /api/audit/verify-chain` 校验整链
- 被改任意一条立刻断链，可定位到具体行号

**预估工作量**：2 人天

---

### 1.3【P2】三权分立的"互斥"再增强

**动机**
PRD 3.2 说"超管不能导出审计报告"，目前是路由级 RBAC。但同账号若被先后赋予 super_admin → auditor，会留下绕过分权的可能。

**改造点**
- 同账号生命周期内不能持有过 `super_admin × auditor` 互斥角色组合
- 在 `users` 表增加 `historical_roles` JSON 字段
- 修改用户角色时校验历史角色互斥规则
- 创建账号时也校验

**预估工作量**：1 人天

---

### 1.4【P1】登录暴力破解保护增强

**动机**
当前是"5 次失败锁定 15 分钟"（账号维度）。但攻击者用同一字典撞 100 个账号，每个账号只试 5 次，仍可绕过；登录响应头一致还可能被用户名枚举。

**改造点**
- IP 维度也要锁：同一 IP 1 分钟内 ≥10 次失败 → 锁 IP 15 分钟
- 登录接口加 captcha：连续失败 3 次后启用（前端弹简单图形 captcha 或滑块）
- 登录页响应头：`X-Frame-Options: DENY`（当前 main.py 是 `SAMEORIGIN`，登录页应该 DENY）
- 失败响应统一时间：账号不存在和密码错误响应时间一致（防 timing attack）

**预估工作量**：2 人天

## 2. 安全能力差异化（区别于通用 CMDB 的杀手锏）

### 2.1【P0】危险端口规则可配置 + 自定义规则引擎

**动机**
现在 `reports.py` 里 `DANGEROUS_PORTS` 是硬编码集合（19 个端口），`HIGH_RISK_ZONES` 也写死。但每家客户的"危险"定义不同：
- 医疗行业：HL7 (2575) 端口暴露危险
- 金融行业：1521 必须封禁
- 教育行业：弱密码服务（21、3306）尤为敏感

**改造点**
- 新建 `risk_rules` 表，schema：
  ```sql
  CREATE TABLE risk_rules (
      id INTEGER PRIMARY KEY,
      name VARCHAR(100),                 -- "DMZ 暴露 RDP"
      enabled BOOLEAN DEFAULT 1,
      port_match VARCHAR(50),            -- "3389" 或 "21,22,23" 或 "1024-65535"
      protocol VARCHAR(10),              -- tcp/udp/any
      zone_match VARCHAR(50),            -- "dmz,office" 或 "any"
      service_regex VARCHAR(255),        -- "Apache/2\\.4\\.49" → CVE 命中
      severity VARCHAR(20),              -- high/medium/low
      action VARCHAR(20),                -- alert/block(预留)
      description TEXT,
      created_at DATETIME,
      updated_at DATETIME
  );
  ```
- 提供规则管理 UI（super_admin / admin）
- 规则可导入导出 YAML，便于跨实例同步、预置行业模板
- 报表引擎改为基于规则匹配，不再硬编码

**这是产品差异化，让安全人员自己定义规则远比内置 19 个端口有价值**

**预估工作量**：5 人天（后端 2 + 前端 2 + YAML 导入导出 1）

---

### 2.2【P0】CVE / 漏洞情报联动

**动机**
nmap 已扫到 `service_name + service_version`，"差一口气"就能告警漏洞。这是相对自动化扫描器（绿盟 RSAS、奇安信天眼等）的"轻量平替"路径，定位明确：**不替代专业扫描器，但给中小团队"看得到风险"的能力**。

**改造点**
- **离线模式**：内置 NVD/CNNVD 子集（每月手动更新 JSON 包），按 `service+version` 查 CVE
  - 数据源：NVD JSON Feeds、CNNVD 数据库
  - 存储：`data/cve_db.json` 或独立 SQLite 表
- **在线模式（可选）**：调 NVD API 2.0、奇安信 ti.qianxin.com、阿里云漏洞库
- 资产详情页新增"风险"Tab，列出端口对应的 CVE + CVSS 评分 + 修复建议
- 报表新增"全网 CVE 风险面板"
- 与 2.1 的规则引擎打通：`service_regex` 命中 = 自动关联 CVE

**v1.1 主打能力，可作为商业版/社区版的差异化分水岭**

**预估工作量**：8 人天（NVD 数据导入 + 匹配逻辑 + UI）

---

### 2.3【P2】资产指纹去重与孤儿 IP 检测

**动机**
当前 diff 用 MAC → IP → hostname+OS 三级匹配，但 NAT 后端、虚拟化场景容易把多台机器识别成一台；反过来一台机器换 IP 后被识别为新资产。

**改造点**
- 增加"指纹相似度"得分：
  - 端口集合 Jaccard 相似度
  - OS 指纹文本相似度
  - hostname 编辑距离（Levenshtein）
- 总相似度 > 阈值（默认 0.8）时强制人工 review
- 报表加"疑似重复资产"清单
- 同时识别"孤儿资产"：MAC 变了但 hostname+端口完全相同 → 提示用户合并

**预估工作量**：3 人天

---

### 2.4【P1】网络区域越权检测

**动机**
当 office 区资产开了 1433/3306（数据库端口），暗示这台机被错配置或被植入后门。这种"越权"在常规扫描中是异常信号。

**改造点**
- 基于"网络区域 × 端口"的策略矩阵（管理员可定义"office 不应有数据库端口"）
- 与 2.1 规则引擎共用 `risk_rules` 表
- 扫描确认后自动产出"越权资产"报表
- 资产详情页增加"合规性"红绿灯

**预估工作量**：2 人天（依赖 2.1 完成）

---

### 2.5【P0】HVV 场景专用 Dashboard

**动机**
HVV（护网行动）期间值守人员要在 30 秒内回答几个核心问题：
- 有几个对外暴露的入口？
- 哪些是高危资产？
- 最近 7 天端口变化？
- 影子资产数量？
- 等保不合规的资产数量？

当前要在多个页面切换才能拼起来。

**改造点**
- 新页面：`/reports/hvv-dashboard`（路由）+ `HvvDashboard.vue`
- 一屏聚合：
  - 区域暴露面饼图
  - 危险端口告警列表（Top 20）
  - 7 天端口变化时间线
  - 影子资产 + 长期 offline 资产计数卡片
  - CVE 风险面板（依赖 2.2）
- 支持一键截图导出 PNG（用于值守日报）
- 大屏模式（可投屏给值班室）

**销售场景强需求，HVV 期间的"日报模式"是中型甲方安全团队的刚需**

**预估工作量**：4 人天

## 3. 数据采集与质量

### 3.1【P1】多源导入（不止 nmap）

**动机**
PRD 写的是"坚持手动 nmap 上传保证准确性"，立场对。但实际客户手里还有：Excel 资产台账、CMDB-Lite 早期版本导出、masscan 结果、防火墙策略导出。强制只接 nmap 等于把 50% 客户拒之门外。

**改造点**
- 抽象 `ImportAdapter` 接口，先支持：
  - Excel/CSV 模板导入（按 PRD 6.1 字段，提供下载模板）
  - masscan JSON（端口扫得快但 service 弱，可与已有 nmap 结果合并丰富）
  - 阜时未来可扩展：FortiGate / Cisco 配置导出
- 仍然保持"人工确认"的原则，所有非 nmap 来源也走 ScanConfirm 流程
- 字段缺失时（如 Excel 没有端口）允许部分导入

**预估工作量**：4 人天

---

### 3.2【P2】nmap 上传的预检与回放

**动机**
上传后用户经常发现 XML 不完整或扫错网段，但批次已写进 `scan_snapshot_items`。当前 `pending` 状态做了一半（数据已入库但未影响资产表），UX 上仍有"已经入库"的心智负担。

**改造点**
- 上传后先进入 `preview` 状态（解析在内存中，不写 scan_snapshot_items）
- 前端展示"预计影响 X 个资产，新增 Y 个，差异 Z 个"
- 用户确认后才正式写库
- 提供"重新上传覆盖"动作，避免用户产生大量废批次

**预估工作量**：2 人天

---

### 3.3【P2】字段必填的"渐进式"放宽

**动机**
当前所有"导入新资产"必填 `business_system / owner / location / importance`。客户实测：批量导入时填字段是最大摩擦点，常常一上午就"填字段"几十次。

**改造点**
- 支持"暂存为草稿"（status=draft），允许字段缺失
- 草稿资产 7 天内必须补齐，否则自动加入"影子资产"报表
- 前端列表加"草稿"过滤器
- 草稿资产在拓扑图上以虚线/灰色显示

**预估工作量**：2 人天

---

## 4. 产品体验与协作

### 4.1【P1】资产责任人通知机制

**动机**
当前"危险端口告警"只在 dashboard 显示，没人主动推送。安全工程师不会每天登录 CMDB 看面板，告警必须主动找人。

**改造点**
- 配置 webhook / 邮件 / 企业微信群机器人 / 钉钉群机器人（自托管场景常见）
- 触发条件可配置：
  - 资产 importance=core 出现新端口
  - DMZ 出现 3389/22 等管理端口
  - CVE 严重度 ≥ 7.0 命中
  - 影子资产连续 30 天未补齐
- **不做工单流转**（坚持 PRD 4.1 不做的边界），仅做单向通知
- 通知内容支持模板化（{asset_no} {ip} {risk_desc}）

**预估工作量**：3 人天

---

### 4.2【P2】资产变更审批（轻量版）

**动机**
当前 admin 改 owner/importance 直接生效。高重要性（importance=core）资产的关键字段变更应有"二次确认"。

**改造点**
- 高重要性资产的关键字段（owner / importance / business_system / network_zone）变更走"申请-审批"流
- super_admin 审批
- audit_logs 关联审批记录
- 前端列表显示"待审批"标记

**预估工作量**：3 人天

---

### 4.3【P1】drawio 拓扑图与资产强联动

**动机**
现在 drawio 集成是单向（LLM 生成 → drawio 编辑），编辑后修改的节点和资产表脱钩——拖个新方块上去无法关联到资产；资产下线了拓扑图也不会变。

**改造点**
- drawio 节点保存元数据（custom attribute）携带 `asset_id`
- 双击节点跳资产详情页（postMessage 通信回主框架）
- 资产下线时，拓扑图节点自动加"已下线"角标（保存版本前生成）
- 拓扑图加载时校验：删除已下线资产的连线，并提示用户

**预估工作量**：4 人天

---

### 4.4【P2】拓扑图导出与分享

**动机**
安全工程师汇报时常需要 PNG/PDF 嵌 PPT。drawio 已支持导出，但要走它的菜单（藏得深）。

**改造点**
- 在 TopologyEditor 顶栏加"导出 PNG / PDF / SVG"快捷按钮
- 调用 drawio postMessage 的 `export` action 一键导出
- 支持"链接分享"：生成只读 token URL（限 24 小时），无登录可看（用于跨部门汇报）

**预估工作量**：2 人天

## 5. 工程基线（不性感但不能少）

### 5.1【P0】速率限制中间件

**动机**
FastAPI 当前没有任何速率限制。`/auth/login` 被穷举、`/scans/upload` 被刷会拖垮 SQLite。这是安全产品自身的"信任成本"。

**改造点**
- 用 `slowapi` 或自实现 token bucket
- 关键端点限流：
  - `/auth/login`：5 次/分钟/IP
  - `/scans/upload`：3 次/分钟/user
  - `/reports/*`：60 次/分钟/user
  - `/topology/generate`：5 次/小时/user（LLM 调用昂贵）
- 限流响应头：`X-RateLimit-Remaining` / `X-RateLimit-Reset`
- 限流命中写入 audit_logs，便于追溯异常调用

**预估工作量**：1.5 人天

---

### 5.2【P1】SQLite 备份策略

**动机**
单文件易毁。WAL 模式只防意外断电，不防误删 / 误改 / 磁盘损坏。

**改造点**
- `scripts/backup_db.py` 调用 `sqlite3 .backup`（在线热备，不阻塞）
- 加密压缩（AES-256-GCM + zstd）写到 `data/backups/cmdb_YYYYMMDD_HHMM.db.zst.enc`
- 保留策略：日备 30 天 + 周备 12 周 + 月备 12 月
- docker-compose 加一个 cron sidecar 容器
- 文档里给 systemd timer 范例
- 提供 `scripts/restore_db.py` 恢复脚本

**预估工作量**：2 人天

---

### 5.3【P1】SBOM 与依赖漏洞自查

**动机**
安全产品自身有依赖漏洞会很尴尬（参考 log4j 事件）。客户审计也会问"你的依赖有没有 CVE"。

**改造点**
- CI 加 `pip-audit` / `npm audit` / Trivy 扫 Docker 镜像
- release 产物随附 SBOM（CycloneDX 格式）：
  - `release/cmdb-lite-v1.0-backend.cdx.json`
  - `release/cmdb-lite-v1.0-frontend.cdx.json`
  - `release/cmdb-lite-v1.0-image.cdx.json`
- 文档里增加"依赖漏洞响应流程"章节

**预估工作量**：2 人天（基于 CI 流程是否已存在）

---

### 5.4【P1】上传文件二次校验（资源耗尽防护）

**动机**
当前 `validate_nmap_xml` 只校验根元素 + defusedxml。但攻击者可上传巨型嵌套 XML（亿级 host 节点）让 libnmap 内存爆，单进程 OOM 即拖垮服务。

**改造点**
- 解析前流式扫一遍 XML 元素数量上限（如 50 万节点封顶），超出直接拒
- 主机数 ≥ 1 万触发"大批次警告"，前端提示用户
- 解析阶段加超时（30 秒），libnmap 卡住自动中止
- 解析在子进程跑（multiprocessing），OOM 不影响主进程

**预估工作量**：2 人天

---

### 5.5【P2】JWT 失效列表（注销不仅仅删 cookie）

**动机**
当前 `logout` 只清 refresh cookie，access_token 在剩余 15 分钟内仍可用。等保审计员会问。

**改造点**
- 内存或 SQLite 维护 `revoked_tokens` 表（只存 jti）
- 鉴权时检查
- 登出和密码修改时把当前 access token 加入黑名单
- access_token 加 `jti` claim
- 黑名单超过 access_token 过期时间后自动清理

**预估工作量**：1.5 人天

---

### 5.6【P2】FastAPI lifespan 迁移

**动机**
差异分析里写"deprecation warnings 从 89 降到 4（剩余为 FastAPI on_event 弃用）"。`on_event` 已弃用，FastAPI 推荐用 lifespan handler。升级到 FastAPI 0.111+ 时会强制要求。

**改造点**
- `main.py` 改为 lifespan 上下文管理器
- 顺便把 `ensure_initial_admin` 移到 lifespan 中

**预估工作量**：0.5 人天

---

## 6. 可观测性与运营

### 6.1【P2】Prometheus 指标暴露

**动机**
客户运维常自带监控（Zabbix/Prometheus/Grafana），希望能把 CMDB 自身指标接入统一看板。

**改造点**
- `/metrics` 端点暴露（admin only）：
  - 登录成功/失败 QPS
  - 扫描批次处理耗时直方图
  - LLM 调用耗时分布
  - 当前危险端口数量（gauge）
  - 当前影子资产数量（gauge）
  - SQLite 数据库大小
- 用 `prometheus-fastapi-instrumentator` 一行集成
- 提供示例 Grafana dashboard JSON

**预估工作量**：1.5 人天

---

### 6.2【P2】LLM 调用降级与熔断

**动机**
现在拓扑生成调云端 LLM 失败就直接报错，用户体验差。

**改造点**
- cloud → ollama 自动降级（参考 `topology_service` 已有的路由逻辑）
- 连续失败 3 次熔断 5 分钟
- 熔断期间直接走 ollama 或返回"上次成功的拓扑图"
- 熔断状态在系统配置页可见

**预估工作量**：1.5 人天
